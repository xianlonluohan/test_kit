# Google C++ Style Guide 常用规范要点

## 概述

本文档旨在为初学者和爱好者，提供一份精炼、实用的 C++ 编程规范指引。

本文所介绍的规范要点，均遵循并源自《Google C++ Style Guide》。我们并非创建一套新规则，而是从该权威指南中，精选出在cpp文件开发中最常用、最能立即提升代码质量的若干核心规范。

目标是帮助您在项目初期就建立良好的编码习惯，理解为什么要遵循这些规范，并掌握如何在您的项目中应用它们，从而编写出更清晰、更健壮且易于协作的代码。

## 匿名命名空间

在`.cpp` 文件里有一些函数、变量或常量，你确定它们只会在本文件内部被使用，永远不会被其他文件调用或访问时，就应该使用匿名命名空间。

它的核心作用是实现“内部链接”，将这些东西“隐藏”起来，避免与其他文件中可能存在的同名标识符发生冲突。这是提升代码模块化和安全性的重要习惯。

匿名命名空间的声明应与具名命名空间的格式相同. 在末尾的注释中, 不用填写命名空间名称:

```c++
namespace {
...
}  // namespace
```

**Arduino 示例**:

```c++
#include <cstdint>

// 作用：将其中的常量、变量和函数的作用域限制在本编译单元（即本 .cpp 文件）内。
namespace { 
constexpr int32_t kValue = 0;
int32_t g_value = 0;

void Test() {
  //...
}
}  // namespace

void setup() {
}

void loop() {
}
```

## 类型转换

使用 C++ 的类型转换, 如 static_cast<>(). 不要使用 int y = (int)x 或 int y = int(x) 等转换方式;

不要使用 C 风格类型转换. 而应该使用 C++ 风格.

- 用 `static_cast` 替代 C 风格的值转换, 或某个类指针需要明确的向上转换为父类指针时.
- 用 `const_cast` 去掉 const 限定符.
- 用 `reinterpret_cast` 指针类型和整型或其它指针之间进行不安全的相互转换. 仅在你对所做一切了然于心时使用.

## `const`用法

我们强烈建议你在任何可能的情况下都要使用 `const`. 此外有时改用 C++11 推出的 `constexpr` 更好。

在声明的变量或参数前加上关键字 `const` 用于指明变量值不可被篡改 (如 `const int foo` ). 为类中的函数加上 `const` 限定符表明该函数不会修改类成员变量的状态 (如 `class Foo { int Bar(char c) const; };`)。

## `constexpr`用法

变量可以被声明成 `constexpr` 以表示它是真正意义上的常量，即在编译时和运行时都不变。函数或构造函数也可以被声明成 `constexpr`, 以用来定义 `constexpr` 变量。

`constexpr`的主要优势是允许在编译期计算，这能用于数组大小定义、模板元编程等场景，而不仅仅是“定义真正的常量”。

## 整型

在嵌入式开发中，可移植性和确定性至关重要。C++ 标准并未对`short`、`int`、`long`等基本整数类型的位宽做出强制规定，其大小由编译器和目标平台共同决定。

例如，int在常见的 32 位架构上为 32 位，但在某些 16 位微控制器上可能仅为 16 位。`long`类型在 Windows 和 Linux 64 位系统上通常差异显著。这种不确定性是嵌入式系统跨平台移植时的主要风险来源之一。

**强制规则**：

为避免因类型位宽隐式变化导致的溢出、数据截断及未定义行为，在嵌入式项目中，必须使用具有明确位宽的整数类型，禁止使用 `short`、`int`、`long`、`long long`及其无符号版本。

**正确做法**：

包含标准库头文件 `<cstdint>`，并使用其中定义的类型：

| 类型 | 说明 |
| :--- | :--- |
| int8_t/ uint8_t | 精确的 8 位有/无符号整数 |
| int16_t/ uint16_t | 精确的 16 位有/无符号整数 |
| int32_t/ uint32_t | 精确的 32 位有/无符号整数 |
| int64_t/ uint64_t | 精确的 64 位有/无符号整数 |

**使用场景与说明**：

- **通用选择**：根据变量需要表示的数据范围选择最小但足够的类型。例如，对于小于 255 的循环计数器，使用 uint8_t；对于传感器采集的 16 位 ADC 值，使用 uint16_t。
- **指针与大小**：表示对象大小或数组索引时，使用 `size_t`。表示指针差值时，使用 `ptrdiff_t`。它们是平台相关的，但能确保存储任何该平台上可能出现的对象大小。
- **对“仅使用int”的修正**：在通用编程中，`int`因其“自然大小”常被推荐用于循环变量。但在嵌入式开发中，为了绝对的确定性和可移植性，即使对于循环计数器，也应使用 `int32_t` 或根据数值范围选择的固定宽度类型。

## 预处理宏

避免定义宏，尤其是在头文件中；应优先使用内联函数、枚举和常量变量。为宏起名时应使用项目特有的前缀。不要使用宏来定义 C++ API 的部分内容。

宏意味着你所看到的代码与编译器所看到的代码并不相同。这可能会导致意想不到的错误，尤其是因为宏具有全局作用域。

幸运的是，在 C++ 中，宏的作用远不如在 C 语言中那么重要。不要使用宏来内联性能关键的代码，而应使用内联函数。不要使用宏来存储常量，而**应使用常量变量**。不要使用宏来“简化”冗长的变量名，而应使用引用。不要使用宏来进行条件编译，千万不要这么做（当然，除了使用 #define 保护符来防止头文件的重复包含之外）。这样做会使测试变得更加困难。

宏能够完成其他技术无法实现的任务，而且在代码库中（尤其是在底层库中）确实能看到它们的身影。而且它们的一些特殊功能（比如字符串化、连接等）并不能通过语言本身来实现。但在使用宏之前，请务必仔细考虑是否有非宏的方法能够达到相同的效果。如果需要使用宏来定义接口，请联系项目负责人请求放宽此规则。

以下这种使用模式能够有效避免使用宏时出现的诸多问题；如果您使用宏的话，应尽可能遵循这种模式：

- 不要在.h 文件中定义宏。
- 在使用宏之前先对其进行定义，并在使用完毕后将其取消定义。
- 不要在替换现有宏之前直接取消定义它并用自己定义的宏来替代；相反，应选择一个可能独一无二的名称。
- 尽量不要使用会生成不均衡的 C++ 结构体的宏，或者至少要详细说明这种行为。
- 尽量不要使用 ## 来生成函数/类/变量的名称。

从头文件中导出宏（即在头文件中定义它们，而在头文件末尾之前不取消定义）是非常强烈不被推荐的。如果确实从头文件中导出一个宏，它必须有一个全局唯一的名称。要做到这一点，必须用包含项目命名空间名称（但要大写）的前缀来命名它。

## 0 和 nullptr/NULL

对于指针使用 `nullptr`，对于字符使用“\0”（而不是 0 这个字面值）。

对于指针（地址值），请使用 `nullptr`，因为这样能提供类型安全性。

使用`'\0'`作为空字符。使用正确的类型会使代码更具可读性。

## auto

用 `auto` 绕过烦琐的类型名，只要可读性好就继续用，别用在局部变量之外的地方。

**典型场景**：

```c++
auto it = my_map.find(key); // 明显是迭代器类型
auto* ptr = dynamic_cast<Derived*>(base_ptr); // 明确需要指针
```

## 文件命名

文件名要全部小写, 可以包含下划线`"_"` 或连字符`"-"`, 依照项目的约定. 如果没有约定, 那么**下划线**更好。

```txt
project_file_name.cpp
```

## 类型命名

类型名称的每个单词首字母均大写, 不包含下划线: `MyExcitingClass`, `MyExcitingEnum`。

## 变量命名

变量 (包括函数参数) 和数据成员名一律小写, 单词之间用下划线连接. 类的成员变量以下划线结尾, 但结构体的就不用, 如: a_local_variable, a_struct_data_member, a_class_data_member_。

### 普通变量命名

**示例**：

```c++
string table_name;  // 好 - 用下划线.
string tablename;   // 好 - 全小写.

string tableName;  // 差 - 混合大小写
```

### 类数据成员

不管是静态的还是非静态的, 类数据成员都可以和普通变量一样, 但要接下划线.

```c++
class TableInfo {
  ...
 private:
  string table_name_;  // 好 - 后加下划线.
  string tablename_;   // 好.
  static Pool<TableInfo>* pool_;  // 好.
};
```

### 结构体变量

不管是静态的还是非静态的, 结构体数据成员都可以和普通变量一样, 不用像类那样接下划线:

```c++
struct UrlTableProperties {
  string name;
  int num_entries;
  static Pool<UrlTableProperties>* pool;
};
```

## 常量命名

声明为 `constexpr` 或 `const` 的变量, 或在程序运行期间其值始终保持不变的, 命名时以 “k” 开头, 大小写混合. 例如:

```c++
const int kDaysInAWeek = 7;
```

## 函数命名

通常情况下，函数遵循**大驼峰命名法 (UpperCamelCase)**：以大写字母开头，并且每个新单词都以大写字母结尾。

```c++
AddTableEntry()
DeleteUrl()
OpenFileOrDie()
```

同样的命名规则也适用于作为 API 部分公开的、旨在看起来像函数的类和命名空间范围内的常量，因为它们是对象而非函数这一事实只是一个不重要的实现细节。

访问器和修改器（获取和设置函数）可以像变量一样命名，采用“下划线分隔”的形式。这些通常与实际的成员变量相对应，但并非必须如此。例如，函数“int count()”和“void set_count(int count)”。

## 枚举命名

枚举变量（无论是作用域内的枚举还是无作用域的枚举）的命名应与常量的命名方式相同，而非与宏的命名方式相同。也就是说，应使用“kEnumName”而非“ENUM_NAME”。

```c++
enum class UrlTableError {
  kOk = 0,
  kOutOfMemory,
  kMalformedInput,
};
```

_直到 2009 年 1 月，之前的做法是将枚举值的命名方式设定为类似于宏的形式。这导致了枚举值与宏之间名称冲突的问题。因此，为了消除这种问题，决定改为采用常量风格的命名方式。新的代码应当采用常量风格的命名方式_。

## 宏命名

通常**不应该**使用宏. 如果不得不用, 其命名像枚举命名一样全部大写, 使用下划线:

```c++
#define MYPROJECT_ROUND(x) ...
```

## 空格还是制表位

只使用空格，并每次缩进 `2` 个空格。

我们使用空格来进行缩进。在代码中不要使用制表符。您应该将编辑器设置为在按下制表键时以空格形式显示内容。

## 条件语句

倾向于不在圆括号内使用空格. 关键字 if 和 else 另起一行.

最常见的是没有空格的格式. 哪一种都可以, 最重要的是 保持一致. 如果你是在修改一个文件, 参考当前已有格式。如果是写新的代码, 参考目录下或项目中其它文件。还在犹豫的话, 就不要加空格了。

```c++
if (condition) {  // 圆括号里没有空格.
  ...  // 2 空格缩进.
} else if (...) {  // else 与 if 的右括号同一行.
  ...
} else {
  ...
}
```

## 总结与进一步学习建议

本文档所阐述的要点，均提炼自《Google C++ Style Guide》这一业界广泛认可的权威编码规范。我们聚焦于在嵌入式开发等场景下最常用、最能立即提升代码质量的核心规则，例如强制使用确定宽度的整型、正确的const用法、明确的命名约定以及避免宏陷阱等，旨在为您提供一个清晰、可快速上手的实践起点。

然而，必须明确指出，**本文档仅是原规范的一个子集和精要导读**。它无法替代阅读完整的原版指南。

**我们强烈建议您在有条件时，直接阅读《Google C++ Style Guide》原文**。这样做的好处包括：

1. **获得完整图景**：原文涵盖了C++语言的几乎所有方面，包括内存管理、类设计、继承、模板、异常、RTTI等高级主题，能帮助您构建系统性的知识体系。
2. **理解规则背后的“为什么”**：原文对每一条规则都提供了详尽的原理说明、利弊权衡和例外情况。理解其设计哲学，远比机械地记忆规则列表更为重要，这能让您在面对未覆盖的特殊情况时做出更合理的判断。
3. **查看丰富的正反示例**：原文配有大量代码示例，直观展示“好代码”与“坏代码”的对比，加深对规则的理解和应用。
4. **适应更广泛的场景**：虽然本文档针对嵌入式场景进行了强化（如整型使用），但原指南面向的是更普遍的软件工程领域。通读原文能让您了解在不同规模、不同需求的项目中，规范可能存在的灵活性及考量。

遵循一套一致的、经过深思熟虑的编码规范，其终极目的是为了提升代码的**可读性、可维护性和团队协作效率**。希望本指南能成为您培养良好编程习惯的第一步，并引导您深入探索更广阔的软件开发最佳实践。
